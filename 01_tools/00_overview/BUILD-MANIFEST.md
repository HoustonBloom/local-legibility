# BUILD-MANIFEST system

The packet's entry file : `Getting Started.html` : is generated by
`build-window-frames.mjs` from source files in `01_tools/01_lenses/`
(welcome, workflow, about) and `01_tools/00_overview/lens-catalog.html`,
plus base64-injected example payloads.

Because the output is a single self-contained HTML file that is sometimes
edited **in place** (you may tune copy directly in the bundled HTML rather
than in the source files), there is a recurring failure mode without
drift protection: a subsequent rebuild silently clobbers inline edits by
regenerating from stale source. The drift-protection system catches this
before a rebuild overwrites your work.

`BUILD-MANIFEST.yaml` (written next to `build-window-frames.mjs` in
`01_tools/00_overview/`) is the fingerprint that makes the drift
detectable. It lives alongside the build tooling, not at packet root,
so downstream users who install the packet see a clean top-level
directory.

## What the manifest records

```yaml
last-built: <ISO timestamp>
output-size: <bytes>
output-sha1: <hex>
frames:
  - { id: frame-welcome, label: "Getting Started", source: "...", source-sha1: <hex> }
  - ...
examples:
  - { token: EXAMPLE_REEL, source: "...", source-sha1: <hex> }
  - ...
```

Every SHA is taken at build time. The build also calls
`fs.utimesSync(OUT, manifestMtime, manifestMtime)` immediately after
writing the output, so the output file's mtime matches the manifest's
`last-built` timestamp to the millisecond. This lets the next build
detect edits-since-last-build by comparing mtimes.

## Preflight behavior

At the start of every build, `preflight()` runs:

1. If no manifest exists and output exists → abort (unknown provenance).
2. If output mtime is more than ~5 seconds newer than `last-built` →
   abort with a warning that someone edited the output directly.
3. Otherwise → proceed with the rebuild.

Aborts can be bypassed with `--force`.

## Reconciling source files from a canonical output

If the output has edits that are not present in source files (i.e., the
real fix is "source files are stale"), do this before rebuilding:

1. For each frame in the manifest, read the iframe's `srcdoc` out of
   `Getting Started.html`, HTML-entity-decode it (`&quot;` → `"`,
   `&amp;` → `&`, etc.), and write the result to that frame's source
   file.
2. Copy the prior source file to `01_tools/01_lenses/_deprecated/` with
   a `.pre-reconcile-YYYY-MM-DD.html` suffix so prior state is
   recoverable.
3. Rerun `node build-window-frames.mjs --force`. The new manifest will
   fingerprint the reconciled sources.

A worked example of this reconciliation ran on 2026-04-21 : see
`outputs/reconcile_sources.py` in session logs for the shape of the
script.

## When to use `--force`

- First build after adding the manifest system (no prior manifest).
- After reconciling source files (step 3 above).
- After intentionally discarding in-place edits.

Never `--force` as a default. The abort is protecting edits you may
not have noticed.

## Registry of tabs

The `FRAMES` array in `build-window-frames.mjs` is the authoritative
list of tabs. The packet currently ships 4 tabs:

| id              | label            | source                               |
| --------------- | ---------------- | ------------------------------------ |
| frame-welcome   | Getting Started  | `01_tools/01_lenses/welcome.html`    |
| frame-lenses    | Lenses           | `01_tools/00_overview/lens-catalog.html` |
| frame-workflow  | Workflows        | `01_tools/01_lenses/workflow.html`   |
| frame-about     | About            | `01_tools/01_lenses/about.html`      |

Earlier revisions included `frame-setup` and `frame-get-started`; those
were dropped in the canonical shape on 2026-04-20 and should not be
re-added without an explicit ask.

## Registry of example payloads

`EXAMPLE_FRAMES` (in the same script) maps placeholder tokens in
`lens-catalog.html` to example HTML files in
`01_tools/01_lenses/_examples/`. Each placeholder is substituted with
the base64 of its file at build time and surfaced as a
`<script type="text/html" id="...">` payload inside the Lenses frame.

Adding a new lens example requires four touchpoints (see
`01_tools/01_lenses/ADDING-A-LENS.md`): the example file itself, an
`EXAMPLE_*` entry here, a lens card + `{{EXAMPLE_*}}` placeholder in
`lens-catalog.html`, and an entry in `verify-window-frames.mjs`.
