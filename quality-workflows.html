Packet Workflows · Will Good Systems

Three skills that make an agent reuse honestly, display deliberately, and rework boldly

A vault that produces artifacts accumulates framing, diagrams, and design decisions. These workflows stop the agent from re-deriving that work at full cost, from defaulting every page to the same two layouts, and from spending thirty prompts on timid edits when the page needs a rebuild. They ship together in the packet and compose into one quality loop, but each stands alone.

Overview

Each skill targets one named failure an agent falls into by default:

artifact-sourcing

Rebuilding what already exists, or reusing what went stale

New artifacts start from a grep of two generated indexes instead of a scan of raw HTML. Everything reused gets verified against its canonical sources first. Freshness is recorded state, not per-session detective work.

artifact-sourcing/SKILL.md

display-palette

Reaching for cards and tables by reflex

One reference lists every presentation pattern the corpus owns, by information shape, with measured usage. The display decision names what was considered, not just what was chosen.

display-palette/SKILL.md

narrative-overhaul

Timid patching when the page needs demolition

"This page isn't working" flips the stance: one checkpoint on the diagnosis and new spine, then a full rebuild with a reader-walkthrough gate. The invocation is the permission.

narrative-overhaul/SKILL.md

All three follow the packet convention: generic instructions, a path adapter mapping to your vault's actual locations, and graceful degradation when an optional piece (contact profiles, a palette doc) is absent.

The loop

Used together, the three form one pass over any new or failing artifact:

  1. 1
    Find what the corpus holds. Grep the catalog by topic or display pattern; grep the text sidecars for content questions. Open only matched sections.
  2. 2
    Verify before reuse. Read the freshness line. STALE means go to canon; verified means skip the re-check; unchecked means open the Inputs sources and confirm. Drift found becomes a stale mark and an offer to fix.
  3. 3
    Choose the display deliberately. Survey the palette, match the information shape, state a one-line verdict naming the alternatives considered.
  4. 4
    Build, or rebuild. Sourced material is the floor, not the shape. If the page exists and fails, narrative-overhaul takes over: contract, diagnosis, spine, one checkpoint, full execution.
  5. 5
    Walk it as the reader. Section by section: what do they know now, what question formed, does the next section answer it. Not done until this runs clean.
  6. 6
    Log the decision. One line in the sourcing log. At ten entries the session-start scan prompts an audit against the changelog record. The system grades itself.

artifact-sourcing

Reuse-over-rebuild with a verification gate. Two generated artifacts make the corpus searchable without reading it: a catalog (one greppable block per artifact: sections with anchors, declared sources, summary, display vocabulary, freshness) and per-artifact text sidecars (reader-visible text with heading anchors, so a content grep maps straight to the exact section of the exact file).

Both regenerate from a write hook. On the reference corpus (296 artifacts, 18.4 MB of HTML) the full build measured 1.8 seconds and the steady state re-extracts exactly one file per edit; the sidecars total 3.4 MB, about a fifth of the HTML they replace as a search surface.

The freshness protocol

  • STALE (with date and reason in the artifact's config): never source the named sections; use canon directly. The catalog rosters every stale artifact at the top as the batch-repair queue.
  • verified <date>: an agent checked this artifact against its Inputs on that date. Newer than the last edit means skip re-verification.
  • (unchecked): verify against the config's Inputs before reuse. The source always wins a disagreement.
Deprecation happens by moving a file, which fires no write hook. After moves, refresh the indexes manually. The catalog's deprecated roster carries explicit DEPRECATED markers so a grep for a superseded artifact hits a warning, never silence.

Delegation threshold

Catalog greps and single-section reads stay in the main session: the builder must read sourced material first-hand for the display decision to mean anything. Subagents take only the bulk work: verifying against four or more sources, batch stale repair, the audit.

display-palette

The full pattern inventory, by information shape. Answers the question "am I using everything available to me in the way that best serves this?" with a measured document instead of the agent's memory.

Building it: measure pattern usage across the live corpus (the sourcing indexes already carry a display field per artifact), then write one row per pattern with the information shape it serves, when it wins, the usage count, and a known-good exemplar to open. The reference corpus made the case plainly: cards appeared in 185 of 296 artifacts and tables in 112, while storyboard (18), reel (17), step rows (9), filter chips (5), and icon chips (3) sat underused. Not because they are weak; because cards are the reflex.

Consulting it

  • Every display pass surveys the palette before choosing. The inherited or habitual format is a candidate, never a default.
  • The verdict is stated in one line and names the alternatives: "considered storyboard, chose table because the reader compares options rather than follows a sequence."
  • An information shape with no owned pattern (commonly charts, maps, before/after) is flagged as a design opportunity, never force-fit.
Maintenance is three one-line edits when a new pattern ships: a detection signal in the index script, a row in the palette, a note in the component vocabulary.

narrative-overhaul

Teardown and rebuild when a page isn't working. The failure it kills: an agent treating "complete narrative overhaul" as a series of polite local edits, preserving broken structure because it exists, and burning thirty prompts getting where one rebuild would have gone. Invocation is the permission; preservation happens by protocol (the old version is kept per the vault's deprecation convention), so there is no excuse for timidity.

The seven steps

  1. Preserve the old version. Never delete.
  2. Reader contract: who this is for, what they know, what they must walk away with, what surface. Register calibration picks the row: the principal's own reference can assume full context; collaborators get terms defined at first use; the public earns every concept before it is used.
  3. Diagnose structurally: a fault list, not fixes. Spine faults, register faults, title decay, voice-rule violations, prose describing what a diagram should show.
  4. New spine, ONE checkpoint: contract, faults, old spine vs new, standalone-title candidates, in a single message. One confirmation, then full execution with no per-section permission asks.
  5. Rebuild fresh against the spine. The old version supplies facts and evidence, never structure or phrasing.
  6. Walk it as the reader, top to bottom, until the walkthrough runs clean.
  7. Verify and log: writing-rules grep at zero hits, titles read alone in order, layout checks, the round recorded.
A page failing register is a page written for the wrong audience row. Checking which row the current draft was actually written for is often the entire diagnosis.

Contracts

The portable part is the shapes, not the paths. Any vault that produces these three record types can run the workflows.

Catalog block (one per artifact)

frames-catalog.md · one block
=== <slug>
date: … | project: … | lens: … | audience: … | tag: …
title: …
summary: …
sections: Heading A [#s1]; Heading B [#s2]; Heading C
display: cards; svg diagrams (4, lightbox); tables (1)
sources: path1; path2; [[wikilink]] (+N more)
freshness: STALE 2026-07-05 (reason) | verified 2026-07-05 | (unchecked)
html: frames/<slug>.html

Freshness fields (artifact config frontmatter)

FieldWritten whenEffect
stale: trueAn agent finds the artifact contradicted by canon or newer session contextCatalog rosters it; its named sections are never sourced
stale-date / stale-reasonWith the stale markOne line naming what changed and which sections
sources-verifiedAn agent checks the artifact against its Inputs and it holdsFuture agents skip re-verification while it outdates the last edit

Sourcing log line (one per artifact built)

_sourcing-log.md · entry format
date | new artifact slug | sourced (slug#anchor, or "none found" / "none relevant")
     | display verdict | stale marks set | grep reformulations (count)

At ten unaudited entries, the session-start scan prompts the audit: iteration rounds per artifact before vs after adoption, the misses column, display verdicts against what shipped.
